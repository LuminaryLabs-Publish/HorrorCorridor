## Source evidence

```txt
createHost.ts
  connection.on(error)
    -> emits peer/error
    -> does not remove connections map record
    -> does not close connection
    -> does not emit connection-close

createClient.ts
  connection.on(error)
    -> sets currentStatus=error
    -> emits peer/error and peer/status
    -> does not clear activeConnection
    -> does not detach predecessor callbacks

peerEvents.ts
  peer/error has role, roomId, peerId, message and timestamp
  peer/error has no remotePeerId, connectionId, generation, scope or terminality

GameShell.tsx
  handles status, connection-open, connection-close and message
  has no peer/error retirement branch
```

## Finding

Connection-level and peer-level failures are observationally conflated. A host can retain an errored connection and guest roster member without a close event. A client can retain an errored active connection and later install a replacement while callbacks from the predecessor remain live. No generation fence prevents late open or close callbacks from mutating newer session state.
