# Agent Memory

Status: active

## Durable Notes

- `.agent/` is the repo-local control plane. Every future run should start by reviewing `start-here.md`, `workflow.md`, `intention.md`, `live-agent.md`, `memory.md`, `goal.md`, `feedback.md`, and then `change-log.md` before broader repo exploration.
- `HorrorCorridor-Harness` is the repo-owned linear Codex CLI harness for refreshing durable process and domain-kit guidance into `docs/`. Use it when a chat introduces stable rules that should survive beyond the current session.
- `horror-corridor-visual-match` is the repo-owned Codex CLI visual acceptance loop. It should generate or register one reference image, capture current player-view screenshots through the live-player harness, ask Codex to score semantic visual deltas, and auto-fix below-90 results through bounded attempts with artifacts.
- The visual-match CLI follows the NexusDaydreamer-Codex pattern: Codex/imagegen output is not trusted until a submit command registers the image into the run folder, and each attempt owns manifests, prompts, reviews, retry plans, and reports.
- The live agent should be cumulative and episodic rather than a single capped pass: screen review -> action selection -> bounded tool loop -> verification -> review -> lesson candidate -> durable promotion only after repeated evidence. It may run indefinitely under supervision, but the reviewer that reads logs/screenshots and writes notes should stay separate from the live loop itself.
- The current risk is accepting render signal-flow proof as visual-quality proof. Future visual passes must be screenshot-driven and fail closed when the frame is unreadable.
- The local `horror-visual-director` skill is now the top-level gate for HorrorCorridor visual work. Use it before any lighting/material/camera/prop/decal/particle/HUD/procedural-world pass so target-vs-current screenshots drive acceptance.
- Global validation skills now enforce that risk: dark/horror scene work must route through `lighting-balance-scout`, and blocked screenshots mean `Unverified`, not fixed.
- HorrorCorridor-specific tuning belongs in `src/protokits/presets/horror-corridor-preset.ts` and content packs, not inside generic domain kits.
- The current scene-dressing layer is visual-only and non-blocking. Collision or gameplay interaction with props should be added later through explicit occupancy/interaction systems.
- `prop-material-fidelity-domain-kit` is the local domain capability for mesh/material profile intent; `sceneDressingRenderer.ts` is the render owner that turns descriptors into Three.js grouped meshes/materials.
- Human-view proof now routes through `npm run validate:live-player` against a running app, preferably after `npm run build` and `npm run start -- --hostname 127.0.0.1 --port 3000`.
- The live-player harness captures `starting-scene.png` before movement and `movement-scene.png` after movement; use the starting frame for spawn composition and the movement frame for locomotion proof.
- Do not mark `SCENE-002` visually fixed until that report is passing and the screenshot is reviewed against the generated target. Current CDP proof passes validation gates, the palette is less green, and pale floor/wall projection slabs have been reduced, but the corridor still reads too wide/warehouse-like with blocky prop silhouettes and weak focal/anomaly route landmarks.
- No-HUD proof must catch icon-only UI too. The current text gate passes while a bottom-left compass/marker remains visible.
- Keep the live-player luminance gate intact; do not lower it to accommodate darker material experiments. Tune renderer/light/material response until `npm run validate:live-player` passes.
- New procedural content should enter through the smoke-first loop: generate descriptors through `/api/procedural-kit-smoke`, validate with `npm run smoke:protokits`, then wire or tune the same descriptor systems in the main `GameCanvas` path and prove with `npm run validate:live-player:dev`.
- Staged loading is now a real `LOADING` screen before `PLAYING`, currently covering maze field, terrain raycast, object kits, PBR materials, and lighting pass.
- The current scene-generation pipeline now has a geometry-aware intermediate graph: triangle surface anchors -> socket graph -> footprint layouts -> scene bundles. This graph is part of the local ProtoKit/preset architecture and should be extended rather than bypassed when replacing random scatter with more authored placement.
- Current visual result after the graph pass: clustered placements are better and smoke/live-player now expose `anchorCount`, `socketCount`, `layoutCount`, and `bundleCount`, but the asset silhouettes are still too primitive and the top roofline/open-sky cap is too flat. The next visual batch should target previewable mesh-profile replacement, not more random density.
- New visual/domain kits should get isolated preview/testing surfaces before they are considered ready for the main corridor. Main-scene promotion should follow preview proof, then smoke proof, then live-player proof.
- Object-kit visual review should use human-context review rooms, not raw canvas frames alone. `npm run review:object-kit -- corridor-lamp` is the first implementation: it captures full-page review-room screenshots with a generated primary target image, supporting references, angle, lighting mode, scale context, checklist, notes, raw canvas support artifacts, and an 8-angle orbit comparison before any lamp promotion decision.
- `corridor-lamp-object-kit` is no longer treated as one monolithic mesh/profile. It composes nine scoped part-profile kits with inline configs: foundation, pole, armature, lamp head, cable/conduit, fastener, material, light, and validation.
- Major world-object families should stay explicit local object kits rather than anonymous descriptor branches. Smoke validation now checks `sceneObjectKitReview` coverage so generated prop kinds, material families, and shared triplanar/PBR shader profiles stay aligned before main-scene promotion.
- Practical corridor landmarks should be paired prop-plus-light systems when appropriate. Lamps are not mesh-only dressing; they should enter as object kits and emit through the shared scene light descriptor path so placement, smoke validation, and player-view lighting stay consistent.
- The harness-generated guide should define the repo vocabulary explicitly: a kit is a composable capability package, while a domain-service kit is a generic domain owner with stable `provides`/`requires`, metadata, and install surfaces that are customized by HorrorCorridor presets and content packs.
